## Summary

`datatug render` reloads a DataTug project from disk and re-saves it. The practical effect is to regenerate the auto-generated `README.md` files inside the project tree without performing a database scan. Useful after upgrading the CLI to a version with improved README rendering.

## Synopsis

```
datatug render
datatug render --project <id>
datatug render --dir <path>
```

## Problem

DataTug projects contain `README.md` files generated by the CLI from the underlying metadata files. When the CLI's renderer changes — new sections, fixed formatting, additional links — the existing `README.md` files inside a project become stale even though the metadata is correct.

A user could re-run `scan` to refresh them, but `scan` requires database connectivity, takes time, and (most importantly) is not safe to run automatically because the database schema may have changed in the meantime. `render` is the safe, fast alternative: it touches only generated files.

## Behavior

### Project resolution

#### REQ: requires-project-context

`render` MUST resolve a project context via the shared CLI conventions ([REQ: project-or-dir-resolution](../README.md#req-project-or-dir-resolution)). If no project context is found, the command MUST exit `3` (NotFound).

### Load-then-save semantics

#### REQ: load-without-scan

`render` MUST load the project via the standard project store and save it back via the same store. It MUST NOT connect to any database, and MUST NOT change any metadata-bearing file content. Only generated artifacts (e.g., `README.md` files) MAY change as a side effect of save.

#### REQ: idempotent

Re-running `datatug render` on a project that has already been rendered with the current CLI version MUST be a no-op on disk (the generated files MUST be byte-identical to the prior render).

### Single-project store

#### REQ: single-project-store

The current implementation uses `filestore.NewSingleProjectStore(storage.SingleProjectID, v.ProjectDir)` to load and save. This MUST remain the path for `render`; multi-project repo handling is out of scope (use a shell loop instead).

## Parameters

| Flag | Aliases | Type | Description |
|---|---|---|---|
| `--project` | `-p` | string | Project ID from the registry. |
| `--dir` | `-d` | string | Project directory path. |

## Exit codes

| Exit code | Meaning |
|---|---|
| `0` | Project loaded and re-saved successfully |
| `3` | Project directory or `--project` ID not found |
| `1` | Generic runtime error (load failure, save failure) |

## Acceptance Criteria

### AC: regenerates-readmes

**Requirements:** render#req:load-without-scan

After manually corrupting a `README.md` inside a project, `datatug render --dir <project>` exits `0` and restores the README to its generated form. Metadata-bearing files are unchanged.

### AC: idempotent-rerun

**Requirements:** render#req:idempotent

Two back-to-back `datatug render` invocations against the same project produce byte-identical project trees after each run.

### AC: never-touches-network

**Requirements:** render#req:load-without-scan

`datatug render` against any project MUST NOT open any network socket. Verifiable by running under `strace`/`dtruss` with no network syscalls expected.

## Open Questions

- Should `render` also be capable of running against a repo-root layout (multiple projects), parallel to [validate REQ: repo-root-multi-project](../validate/README.md#req-repo-root-multi-project)?
- Should there be a `--check` flag that exits non-zero if a render would change any file, suitable as a CI guard against forgetting to commit re-rendered READMEs?
- The implementation prints a `Rendering project: ...` line via `fmt.Printf` without a trailing newline. Should this become a clean structured log line?
